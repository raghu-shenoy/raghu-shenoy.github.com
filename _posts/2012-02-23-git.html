"I'm an egotistical bastard, and I name all my projects after myself. First Linux, now git."-Linus Torvalds
(Note that Torvalds did not in fact name Linux.)
That's what Linus Torvalds, the developer of Linux and Git had to say when he was asked why did he name the free, fast and efficient Version Control System(VCS), which he developed, as GIT . The Reason
I guess most of us have worked with VCS at some point or the other. I had previously worked with a Subversion client to collaborate code with my team mates. In the beginning we ate up a lot of each others code(due to bad commits) as all of us where pretty new to the concept of VCS. There were a lot of sparks flying around and a lot of over writes being swept under the carpet. Finally we had a sit down with our boss who then gave us a crash course on  how to "diff" files before a commit, how to check out and so on. Things went on pretty well after that. But the biggest problem(Guilty Pleasure) which I found working with SVN was the Bottleneck. Now in the SVN case, the "single" server was the bottle neck, the single point of failure. The server goes kaput the team goes on a tea break,sometimes a very long break. By kaput I mean the network goes down or some other similar problem due to which the code is temporarily not reachable. But say the code goes KAPUT(fried), and you have not backed up the data you as well might go and find a quiet corner to cry. The Solution
Therefore when I realised this issue(I don't wanna be a cry baby) I started looking around for some options. It was at this point I came across GIT. The DISTRIBUTED CVS. This looked to me as the second coming because, 
  • Git is distributed.This means there are so many repositories that even if 1 fails,even if it is the sole remote repository hosted on github.com, I can still retrieve it from some other machine or my own machine and continue working. No more crying!
  • The reason why there are so many repositories is because every machine handling that repository has once local repository. Yes you heard it right. No more network lag. This also means you can still work on git while you are offline and commit code to your local directory in case you have no access to the Internet/intranet. 
These were some of the reasons why I temporarily put aside learning the beauty of UNIX as my primary focus and started off with understanding GIT. In case you want to try out git you can follow these steps
  1. The first thing you need to do is register yourself at https://github.com/. The best part is its free*(How much we all love this term.This truly is the barrier breaker among cultures). *At least the public repositories.[And how we hate the fine print]
  2. Once you have signed up you'll be directed to a page which will explain to you in detail how to go about setting up your very own repository(both local and remote), the basic transactions between the local repository and the remote repository such as committing,pushing and fetching code.
 One thing to be noted here before you actually start playing around with the repository you need to understand the whole procedure of how GIT works.
In git a file has to go through a few states before it is actually safe,
  • When you create a new file it will be un-tracked. This means git just knows a file is present in the local working directory which it was previously unaware of.
  • You then have to "add" the file to the Git which is like saying, "OK Mr. Git please mark and add this file so that when I commit, this particular file also gets committed."
  • Then there's a "commit". This means you have committed your files to the local repository and you are safe.
  • Finally you can push all your committed files to the remote repository and share with everyone your amazing piece of code.
You can get a more detailed explanation here : Git Basics
The other amazing thing about git is branching. With branching you can totally freak out on the code because you can branch out from the main code and all the changes you put in will be local to your branch and you wont be messing up any code written currently by others. Branching takes very less system resources in terms of space and also the time required for creating it.
